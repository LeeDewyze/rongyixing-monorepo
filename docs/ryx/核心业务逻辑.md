# 核心业务逻辑

## 产品定位

**企业差旅管理（TMC）移动客户端 — 融易行**

面向企业员工的因公出行场景，提供机票、酒店、火车、用车等预订能力，并对接差旅申请单与审批流程。

## 主流程概览

```
登录 → 选择公司/Agent → 首页选产品 → 搜索 → 选乘客/填信息 → 下单 → 支付 → 订单管理
                ↑
         差旅申请单 / 审批（合规校验）
```

## 核心服务：`TmcService`

路径：`src/app/tmc/tmc.service.ts`

主要职责：

| 能力         | 说明                                                  |
| ------------ | ----------------------------------------------------- |
| 基础资源缓存 | 国内/国际机场、火车站（带 `LastUpdateTime` 增量更新） |
| 多租户       | 当前 TMC 公司、Agent 切换与缓存                       |
| 工作台       | Workbench 数据按 HR 维度加载                          |
| 差旅申请单   | TravelForm 关联、合规提示                             |
| 乘客证件     | 与 `MemberService` 协作加载员工证件                   |
| 支付         | 与 `PayService` 协作                                  |

## 因公 / 因私贯穿流程

legacy 中的因公 / 因私不是一个单独全局字段，而是由入口路由、员工预订权限、出差单上下文、订单提交参数共同决定。

### 1. 入口与路由域

| 场景 | 首页 / 入口 | 产品路由前缀 | 含义 |
| ---- | ----------- | ------------ | ---- |
| 因公 | `tab-tmc-home` / `tab-tmc-home_ryx` | `tmc-*` | 企业差旅 / TMC 流程 |
| 因私 | `tab-public-home` / `tab-public-home_ryx` | `public-*` | 个人出行 / public 流程 |

`tmc.guard.ts` 会根据登录身份的 `Numbers` 判断首页归属：

- `Numbers.TmcId` / `Numbers.AgentId`：进入 TMC/因公首页，并 `hrService.setRoleSource("tmc")`。
- 非 TMC/BPM 登录态：进入 public/因私首页，并 `hrService.setRoleSource("public")`。
- `Numbers.BpmId`：进入 BPM 首页。

`HrService.roleSource` 只表示当前首页角色来源，主要用于首页和导航上下文，不等同于最终订单的 `TravelType`。

### 2. 员工预订权限

`HrService.isSelfBookType()` 读取 `HrApiUrl-Staff-Get` 返回的 `Staff.BookType`：

- `StaffBookType.Self`：本人预订。legacy 会自动补当前登录人为乘客，并限制自由加乘客。
- `StaffBookType.All` 或无 `BookType`：可代订 / 可选择多人。

这个字段表达的是“谁能订”，不是“因公 / 因私”。迁移时不要把 `BookType.Self` 当成因私标识。

典型行为：

- `tmc-flight.service.ts` 的 `checkOrAddSelfBookTypeBookInfo()` 会在 Self 且无乘客时自动创建本人 `BookInfos`。
- `tmc-flight.service.ts` 的 `checkIfCanPassenger()` 在 Self、出差单、改签等场景下禁止继续加乘客。

### 3. 出差单上下文

从差旅申请单进入机票 / 酒店 / 火车时，legacy 会先清空对应产品的 `BookInfos`，再写入带 `travelFormId` / `travelNumber` 的乘客占位，然后跳转到对应 `tmc-*` 搜索页。

进入搜索页后，如果检测到 `travelFormId`：

- 不允许普通加乘客 / 删除 / 重选。
- 可按出差单乘客和行程约束继续预订。
- 因公属性不只体现在 `TravelType`，还体现在出差单绑定与可操作范围。

### 4. 订单提交参数

因公最终在订单 payload 中落到乘客级字段：

- `Passenger.TravelType = OrderTravelType.Business`
- `Passenger.TravelPayType = 当前选择的 OrderTravelPayType`
- 同时携带组织、成本中心、审批人、出差单、外部编号等信息。

典型代码：

- 火车 TMC 预订页默认 `combineInfo.travelType = OrderTravelType.Business`，提交时写入 `p.TravelType` / `p.TravelPayType`。
- 酒店 TMC 预订页同样默认 `OrderTravelType.Business`，提交时写入酒店乘客 DTO。
- public 火车预订页默认 `orderTravelPayType = OrderTravelPayType.Person`，表现为个人支付链路。

### 5. H5 迁移建议

H5 中建议拆成四类状态显式管理：

| 状态 | 建议字段 | 来源 | 用途 |
| ---- | -------- | ---- | ---- |
| 业务域 | `channel: "business" \| "personal"` | 首页 tab / 路由 / 产品入口 | 决定走 TMC 还是 public 流程 |
| 预订权限 | `bookPermission` | `Staff.BookType` | 决定本人预订、代订、是否可加乘客 |
| 出差单上下文 | `travelFormContext` | 出差单入口 / URL / 本地流程态 | 决定乘客锁定、行程约束、审批链路 |
| 下单参数 | `orderTravelType` / `orderTravelPayType` | 预订页选择与默认值 | 最终写入订单 payload |

不要把以上四类状态揉成一个全局 `travelType`。尤其要避免：

- 把 `Staff.BookType.Self` 误判为因私。
- 从 public 路由进入却复用 TMC 的 `OrderTravelType.Business`。
- 有出差单时只保存乘客，丢失 `travelFormId` / `travelNumber` 导致后续可编辑范围错误。

## 业务域详解

### 1. 身份认证

- 密码 / 手机号 / 设备登录
- 登录成功后保存 Ticket，再请求 WebSocket URL
- 未登录或 Ticket 失效跳转 `login`

### 2. 多租户

- 用户可绑定多个 TMC 公司
- `tmc-switch-company` 切换当前操作主体
- Agent 模式有独立 Guard（`agent.guard.ts`）

### 3. 差旅预订

| 产品     | 模块                      | 典型流程                               |
| -------- | ------------------------- | -------------------------------------- |
| 机票     | `tmc-flight`              | 选城市 → 搜航班 → 选舱 → 填乘客 → 下单 |
| 酒店     | `tmc-hotel`               | 选城市 → 选日期 → 列表 → 详情 → 预订   |
| 国际酒店 | `tmc-international-hotel` | 同上，国际资源                         |
| 火车     | `tmc-train`               | 选站 → 搜车次 → 选座 → 下单            |
| 用车     | `tmc-car`                 | 租车 / 开放平台用车                    |

### 4. 差旅合规

- 预订前可关联 **差旅申请单**（TravelForm）
- 超标或需审批时走 `tmc-approval-task`
- 政策校验接口（如 `HOME_POLICY`）返回是否违规

### 5. 乘客管理

- 从企业员工列表选择乘客
- 校验证件类型与号码（`member/pipe/credential.pipe`）
- 支持多人预订

### 6. 订单与支付

1. `tmc-confirm-information` — 确认订单信息
2. 创建订单 → 唤起支付
3. `tmc-checkout-success` — 支付成功页
4. `order/` — 历史订单列表与详情

## 底部 Tab 信息架构

企业主线（`isTmcHome`）：

| Tab    | 旧路由模块         | 说明                       |
| ------ | ------------------ | -------------------------- |
| 首页   | `tab-tmc-home_ryx` | 产品入口、Banner、快捷预订 |
| 待出行 | `tab-tmc-trip_ryx` | 行程列表                   |
| 我的   | `tab-tmc-my_ryx`   | 个人中心、设置             |

另有游客首页（`isPublicHome`）、BPM 壳（`isBpmHome`）等变体，企业主线以 `isTmcHome` 为准。

## 基础数据 API（Resource 类）

| Method                                        | 用途            | 调用方（Service）       |
| --------------------------------------------- | --------------- | ----------------------- |
| `TmcApiHomeUrl-Resource-Airport`              | 国内机场 / 城市 | `TmcService`            |
| `TmcApiHomeUrl-Resource-InternationalAirport` | 国际机场        | `TmcService`            |
| `TmcApiHomeUrl-Resource-TrainStation`         | 火车站          | `TmcService`            |
| `TmcApiHomeUrl-Resource-DomesticHotelCity`    | 国内酒店城市    | `TmcHotelRyxService` 等 |

客户端拉取后写入 Ionic Storage，按 `LastUpdateTime` 增量更新。酒店城市**不由** `TmcService` 加载，详见 [酒店模块.md](./酒店模块.md)。
